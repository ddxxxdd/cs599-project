## 一、新布局设计

### 1.1 整体架构（三栏式）

```
┌─────────────────────────────────────────────────────────────────┐
│                          顶部状态栏                              │

└─────────────────────────────────────────────────────────────────┘
┌────────┬─────────────────────────────────────┬──────────────────┐
```

### 1.2 关键布局变化

**原布局**：
- 左侧：文档列表（固定显示，无法隐藏）
- 中间：历史卡片（嵌套：历史列表 + 回答详情）
- 右侧：文档详情 + 知识库依据

**新布局**：
- **左侧（300px）**：对话历史列表（最新在上，点击切换当前对话）
- **中间（flex-1）**：主聊天区（上：回答展示 / 下：输入区+示例）
- **右上（396px，45%）**：OSS 知识库列表（搜索+点击弹窗查看详情）
- **右下（396px，55%）**：知识库依据（**按相关度降序排列**）

## 二、功能完善详情

### 2.1 检索优化（top-7 → top-5）

**后端修改**：
1. `agent.py`：
   - `CONTEXT_CHUNK_LIMIT = 7 → 5`（最终送入提示词和前端展示的片段数）
   - 新增 `RETRIEVAL_CANDIDATE_K = 8`（检索候选池保持 8，供工具重排）
   - `_synthesize()` 方法：截断 top-5 后**按相关度降序排序**，引用编号 `[1]~[5]` 现在严格递减

2. `retrieval.py`：
   - `search()` 默认 `top_k` 参数从 4 改为 5

**前端修改**：
3. `main.tsx`：
   - 新增 `sortCitationsByRelevance()` 函数：将 citations 按 score 降序排列，**保留原始编号**（避免回答正文中 `[n]` 跳转失效）
   - "知识库依据"面板调用此函数，展示顺序为：相关度最高 → 最低
   - 默认展开前 2 条改为"相关度最高的前 2 条"

**测试同步**：
4. `test_agent.py`：
   - 引用数断言 7→5
   - 新增测试 `test_citations_are_sorted_by_relevance_score`

### 2.2 代码完善（不影响功能）

**性能优化（`retrieval.py`）**：
- 预计算 `doc_counters`（词频 Counter）、`doc_freq`（文档频率）、`vector_norms`（向量模长）
- `chunk_by_id` 索引提前构建，避免每次查询遍历

**健壮性增强**：
- `retrieval.py`：
  - 空查询/非法 top_k 防护
  - 向量维度不匹配（换 Embedding 模型未重建索引）跳过并 warn
  - 向量索引含过期 chunk_id 不再 KeyError
- `llm.py`：LLM 响应缺 `choices`/`content` 时明确报错
- `api.py`：SSE 流异常转为 `final` 事件返回前端，不再静默断流
- `main.tsx`：
  - 单个 SSE 事件解析失败不中断整条流
  - localStorage 满/禁用时静默降级
  - 中文输入法选词回车（`isComposing`）不触发发送

**安全/带宽优化（`agent.py`）**：
- 新增 `_public_state()`：SSE 的 `meta`/`final` 事件不再把完整 prompt 和 retrieved 原文发给浏览器（前端未使用，纯减负）

### 2.3 界面交互增强

**左侧对话历史**：
- 最新对话显示在最上方（`[...messages].reverse()`）
- 每条历史显示：问题标题 + 状态 + 意图标签 + 引用数 + 时间
- 失败对话红色边框，当前选中蓝色高亮
- 无历史时显示空状态提示

**中间聊天区**：
- 用户问题气泡右对齐（蓝色背景）
- 回答卡片左对齐（左下圆角直角样式，区分角色）
- 示例问题按钮点击后填入输入框，发送后仍保留（方便连续测试）

**右上知识库列表**：
- 搜索框支持过滤（标题/分类/标签/API/错误码）
- 列表项显示：彩色徽章（故障/权限/其他）+ 标题 + 分类 + 关联 API + 常见错误
- 点击列表项**弹窗**显示文档详情（分类/摘要/标签/章节折叠面板/官方链接）
- 弹窗支持关闭，不影响主界面

**右下知识库依据**：
- **按相关度从高到低排序**（视觉上更符合重要性）
- 每条依据显示：`[n]` 编号 + 标题 + 分类标签 + 相关度/工具命中标签
- 回答中点击 `[n]` 自动滚动到对应依据并高亮 1.8 秒
- 默认展开相关度最高的 2 条

**响应式设计**：
- 宽屏（>1180px）：三栏布局
- 中屏（860~1180px）：左侧历史+中间聊天纵向堆叠，右侧两卡片纵向排列
- 窄屏（<860px）：左侧历史横向网格（最多 2 列），中间/右侧全宽纵向堆叠

## 五、使用方式

### 5.1 启动项目

```bash
# 1. 确保前端已构建（仅需一次）
cd frontend && npm run build

# 2. 启动后端（自动托管前端构建产物）
cd ..
uv run uvicorn aliyun_oss_rag.api:app --reload
```

访问 http://localhost:8000

### 5.2 核心交互流程

1. **提问**：在中间输入框输入 OSS 技术支持问题（或点击示例），点击"发送"
2. **查看回答**：中间聊天区显示回答，引用编号 `[1]~[5]` 按相关度从高到低
3. **验证依据**：
   - 右下"知识库依据"面板自动按相关度降序排列
   - 点击回答中的 `[n]` 按钮，自动滚动到对应依据并短暂高亮
4. **切换历史**：左侧点击历史对话，中间/右侧立即切换为对应内容
5. **浏览知识库**：
   - 右上搜索框过滤文档（如搜"403"、"STS"、"CORS"）
   - 点击文档项，弹窗显示完整内容（章节可展开）

### 5.3 典型场景

**场景 A：快速排查错误**
1. 输入"OSS 403 AccessDenied 怎么排查"
2. 回答正文显示步骤，引用 `[1]` 相关度最高
3. 点击 `[1]` 跳转到右下依据，确认来源为"OSS 常见错误排查指南"

**场景 B：对比历史方案**
1. 先问"RAM Policy 和 Bucket Policy 的区别"
2. 再问"STS 临时凭证怎么用"
3. 左侧历史列表点击第一条，回答区立即切回，方便对比

**场景 C：深入研究文档**
1. 右上搜索"生命周期"，过滤出相关文档
2. 点击"OSS 生命周期配置最佳实践"
3. 弹窗显示完整章节（概述/配置步骤/成本优化），展开阅读

## 八、常见问题

**Q1：为什么不把历史放在右侧？**  
A：主流聊天应用（ChatGPT/Claude）均将历史放在左侧，用户已形成操作习惯；右侧留给辅助信息（工具/依据），符合从左到右的主次层级。

**Q2：知识库依据的相关度分数怎么解释？**  
A：0~1 浮点数，越高越相关；向量检索（余弦相似度）和 BM25 混合计算后归一化得出；分数 ≥100 或 ≤0 视为"工具命中"（非检索召回）。

**Q3：为什么输入框放在中间底部，而不是顶部？**  
A：主流对话应用均采用底部输入（模拟移动端消息应用），用户视线自然落在最新内容附近，减少鼠标/视线移动距离。

**Q4：测试显示 1 个 warning（StarletteDeprecationWarning），影响功能吗？**  
A：不影响。这是 FastAPI TestClient 依赖升级提示，运行时功能完全正常。
